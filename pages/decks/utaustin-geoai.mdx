## The Old Way

**Traditional SDI**

- Centralized servers & databases
- WMS / WFS / WMTS protocols
- Government-centered, siloed
- Fragile — one server down = data gone
- Download entire files to use a single tile

</div>
<div>

## The New Way

**Cloud-Native**

- Object storage (S3, GCS, Azure Blob)
- HTTP range requests — read only what you need
- Streamable, composable, interoperable
- Ecosystem-driven (open specs, open data)
- No servers required — just URLs

</div>
</Columns>

<SpeakerNotes>
The shift from traditional SDI to cloud-native geospatial is as fundamental
as the shift from FTP to the web. The key insight: data should be accessible
via simple HTTP requests, not locked behind proprietary services.
</SpeakerNotes>

# The Cloud-Native Geospatial Stack

| Format | What It Does | Key Feature |
|--------|-------------|-------------|
| **COG** | Cloud-Optimized GeoTIFF | Raster imagery via HTTP range reads |
| **STAC** | SpatioTemporal Asset Catalog | Metadata search & discovery |
| **GeoParquet** | Vector data at scale | Columnar, now in core Parquet spec |
| **Zarr** | Multidimensional arrays | Weather, climate, time series |
| **PMTiles** | Serverless vector/raster tiles | Single-file archive, no tile server |
| **COPC** | Cloud-Optimized Point Clouds | LiDAR data via HTTP range reads |

<SpeakerNotes>
Reference Chris Holmes' "Towards a Global Spatial Data Ecosystem" — this stack
already exists and is production-ready. COGs are the workhorse for imagery,
STAC is the catalog layer, GeoParquet handles vector at scale, Zarr covers
n-dimensional data, PMTiles eliminates tile servers, and COPC does for point
clouds what COG did for rasters.
</SpeakerNotes>

# The Ecosystem Already Exists

- **2.6 billion buildings** — Overture Maps Foundation
- **119 million field boundaries** — Fields of The World (FTW)
- **100+ million places** — Overture Maps
- **Petabytes of weather & climate data** — ERA5, CMIP6 in Zarr
- **Trillions of point cloud points** — USGS 3DEP, national surveys
- **Hundreds of millions of embeddings** — Clay, SatCLIP, GeoCLIP

> All streamable directly from object storage. No servers needed.

<SpeakerNotes>

is largely solved — now we need better tools to use it.

# The GeoAI Toolbox

- **TorchGeo** — datasets, samplers, transforms for geospatial ML (PyTorch)
- **SMP** (segmentation-models-pytorch) — encoder-decoder architectures for dense prediction
- **Rasterio / GDAL** — raster I/O backbone, the bedrock of geospatial Python
- **GeoPandas / DuckDB** — vector analysis at scale, SQL on GeoParquet
- **Planetary Computer / Earth Engine** — data access APIs for petabyte archives

> These tools compose naturally with cloud-native formats — read COGs with Rasterio, query STAC catalogs, analyze GeoParquet with DuckDB.

<SpeakerNotes>
TorchGeo and SMP are the two libraries I maintain. TorchGeo handles the
geospatial-specific parts of ML (sampling, CRS alignment, multispectral data)
while SMP provides the model architectures. Together with Rasterio for I/O
and DuckDB for vector queries, you have a complete stack.
</SpeakerNotes>

# How GeoAI Models Process Large Scenes

**The challenge:**

- Satellite scenes are massive — 10,000+ x 10,000+ pixels
- Models expect small tiles — 224x224 or 512x512
- Need to tile, predict, reassemble
- Handle overlaps, CRS alignment, nodata values
- Output must be georeferenced — not just pixel arrays

> This is where TorchGeo's GeoSampler abstraction shines.

<SpeakerNotes>
This is a fundamental challenge in geospatial ML that doesn't exist in natural
image processing. A single Sentinel-2 scene is ~10980x10980 pixels across 13 bands.
You can't feed that to a model — you need to tile it, run inference on each tile,
and stitch the results back together with proper georeferencing.
</SpeakerNotes>

# TorchGeo: The Training Pipeline

```python
from torchgeo.datasets import Sentinel2, EnviroAtlas
from torchgeo.samplers import RandomGeoSampler
from torch.utils.data import DataLoader

# Intersection dataset: imagery + labels
dataset = Sentinel2("data/") & EnviroAtlas("labels/")

# Sample random 256x256 patches from the intersection
sampler = RandomGeoSampler(dataset, size=256, length=1000)

# Standard PyTorch DataLoader with geo-aware batching
dataloader = DataLoader(dataset, sampler=sampler, batch_size=16)
```

<SpeakerNotes>
TorchGeo's key innovation is treating geospatial data as continuous raster fields
indexed by space and time, not discrete files. The ampersand operator creates an
IntersectionDataset that only samples from areas where both imagery and labels exist.
GeoSampler handles CRS alignment, resolution matching, and spatial indexing
automatically. This is standard PyTorch — just with geospatial awareness built in.
</SpeakerNotes>

## Inference at Scale

**The Pattern**

1. Sliding window over COG (HTTP range reads)
2. Batch inference on GPU
3. Write predictions as new COG or GeoParquet
4. Register output in STAC catalog

> Input is cloud-native. Output is cloud-native. No intermediate files on disk.

</div>
<div>

## Cloud-Native Advantage

- **No download needed** — process in place
- **Stream only the tiles you need** — skip nodata regions
- **Output is immediately accessible** — new COG, new STAC item
- **Horizontal scaling** — partition by tile, run on multiple GPUs
- **Reproducible** — input URLs + model version = deterministic pipeline

</div>
</Columns>

<SpeakerNotes>
The cloud-native advantage for inference is huge. Traditional workflows download
entire scenes to disk, process them, then upload results. Cloud-native workflows
read only the tiles they need via HTTP range requests, run inference, and write
results directly to object storage. No intermediate I/O, no disk management.
</SpeakerNotes>

# AI x Cloud-Native Geo = Virtuous Cycle

- **Better data → better models → better tools → more data**
- Active learning: human edits improve model training data
- Common schemas (Overture, FTW) enable cross-dataset benchmarking
- AI coding agents accelerate tool development (Claude, Codex)
- Open formats mean anyone can contribute and consume

> The flywheel is spinning. Cloud-native data makes AI training easier. AI makes cloud-native tools better. Repeat.

<SpeakerNotes>
This is the thesis of the talk: cloud-native geospatial and GeoAI aren't separate
trends — they're a feedback loop. Better data accessibility makes it easier to train
models. Better models produce better data products. Better tools (built faster with
AI agents) make both more accessible.
</SpeakerNotes>

# What's Next

- **Multimodal foundation models** — SAR + optical + text in a single encoder
- **Real-time inference** on streaming satellite data

- **The spatial data ecosystem is the substrate** — AI is the catalyst
- **Agentic workflows** — Claude reads a STAC catalog, writes the inference pipeline, deploys the results
