## How COGs Actually Work

- Standard GeoTIFF + **internal tiling** + **overviews**
- HTTP **range requests** → read just the pixels you need
- No server logic — works with any object storage

> The file *is* the API.

</div>
<div>

<Video src="/videos/S05_COGByteRange.webm" title="COG Byte Range Requests" />

</div>
</Columns>

<SpeakerNotes>
The key insight: a COG is just a regular TIFF file reorganized so that
byte ranges map to spatial tiles. Any HTTP server that supports Range
headers becomes a tile server for free.
</SpeakerNotes>

## GeoParquet at Scale

- Columnar format with **row groups**
- Spatial statistics enable **predicate pushdown**
- Skip non-matching data without reading it
- DuckDB queries 2.6B buildings from your laptop

<SpeakerNotes>
GeoParquet stores data in columnar row groups with spatial statistics.
When you query with a bounding box, DuckDB can skip entire row groups
that don't overlap your AOI — predicate pushdown.

# Query 2.6B Buildings with DuckDB

```python
import duckdb

duckdb.sql("""
    SELECT type, height, ST_Area(geometry) as area
    FROM read_parquet(
        's3://overturemaps-us-west-2/release/2025-03-19.0/
         theme=buildings/type=building/*'
    )
    WHERE bbox.xmin > -97.8 AND bbox.xmax < -97.7
      AND bbox.ymin > 30.2  AND bbox.ymax < 30.3
    LIMIT 100
""")
```

<SpeakerNotes>
DuckDB reads GeoParquet directly from S3 — no download, no database server.
</SpeakerNotes>

Like pretraining LLMs on all internet text, the goal is to **build a generalist geospatial model that can process all modalities** and then either produce embeddings or be fine-tuned downstream.

# What Doesn't Work Yet

- **No single model dominates** — GEO-Bench shows no FM consistently beats task-specific baselines
- **Geographic bias** — Prithvi v1.0 was US-only; Clay and DOFA have better global coverage but still patchy
- **Resolution loss** — MAE-based ViTs downsample 4–5x (224px input → 14×14 patches)
- **Mostly optical** — SAR and hyperspectral remain underserved (DOFA is the exception)
- **Supervised baselines still win** on well-labeled regional tasks
- **AI agents hallucinate** CRS and projection bugs — always verify spatial metadata

<SpeakerNotes>
Honest assessment. The hype cycle is real but so are the limitations.
GEO-Bench (Lacoste et al. NeurIPS 2023) is the most rigorous comparison so far.
Nature Machine Intelligence 2025 survey highlights geographic bias and compute
barriers as the top obstacles to operational deployment.
</SpeakerNotes>

**Contrastive Learning**

</div>

<Video src="/videos/S08_ContrastiveLearning.webm" title="Contrastive Learning on Satellite Tiles" />

</Columns>

<SpeakerNotes>
MAE learns spatial structure by predicting missing patches. Contrastive learning
learns what makes tiles similar or different. They produce different kinds of
features — MAE for dense prediction, contrastive for retrieval.
For MAE, 75% of patches are masked; encoder sees only visible 25%, decoder
reconstructs the full image. Two augmented views of the same satellite tile get
encoded; embeddings pulled together, different tiles pushed apart.
</SpeakerNotes>

# Pixel vs Patch Embeddings

<Columns cols={2} gap={4} align="center" ratio="3fr 2fr">
<div style={{ fontSize: '0.01em', lineHeight: 0.8 }}>

| | **Patch Embedding** | **Pixel Embedding** |
|---|---|---|
| **Output shape** | One vector per patch (e.g. 16x16) | One vector per pixel |
| **Spatial detail** | Compressed — summarizes patch | High — preserves pixel-level info |
| **Best for** | Classification, retrieval, generation | Segmentation, change detection |
| **Context window** | Wide — entire patch context | Narrow — local pixel neighborhood |
| **Examples** | SatMAE, Clay, DOFA, Prithvi | Presto, SatCLIP, AlphaEarth, TESSERA |

> Most foundation models are moving toward **pixel embeddings** with vision transformers — the spatial compression is worth the richer semantics.

</div>
<div>

<Video src="/videos/S10_PixelVsPatchEmbedding.webm" title="Pixel vs Patch Embeddings" />

</div>
</Columns>

<SpeakerNotes>
An important design choice: pixel embeddings (Presto, SatCLIP) produce one
vector per pixel — great for dense tasks like segmentation. Patch embeddings
(SatMAE, Clay, DOFA) compress spatial info into one vector per patch — richer
semantics, better for classification and retrieval.
</SpeakerNotes>

# TorchGeo: Sampling & Training

<Columns cols={2} gap={4} align="center">
<div>

- Geospatial data = **continuous raster fields**
- `&` operator creates **IntersectionDataset**
- Only samples where imagery + labels overlap
- Handles CRS, resolution, alignment automatically

<Video src="/videos/S03_TorchGeoSampling.webm" title="TorchGeo GeoSampling" />

</div>
<div>

<div style={{ fontSize: '0.68em', lineHeight: 1.25 }}>

```python
from torchgeo.datasets import Sentinel2, EnviroAtlas
from torchgeo.samplers import RandomGeoSampler
from torch.utils.data import DataLoader

dataset = Sentinel2("data/") & EnviroAtlas("labels/")
sampler = RandomGeoSampler(
    dataset,
    size=256,
    length=1000,
)
dataloader = DataLoader(
    dataset,
    sampler=sampler,
    batch_size=16,
)

for batch in dataloader:
    x, y = batch["image"], batch["mask"]
    y_pred = model(x)
    loss = criterion(y_pred, y)
    loss.backward()
    ...
```

</div>

</div>
</Columns>

<SpeakerNotes>
TorchGeo treats geospatial data as continuous raster fields indexed by space
and time. The ampersand operator creates an IntersectionDataset.
The intersection operator is TorchGeo's key innovation. Standard PyTorch
DataLoader — just with geospatial awareness built in.
</SpeakerNotes>
