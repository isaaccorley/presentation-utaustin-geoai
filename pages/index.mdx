# Query 2.6B Buildings with DuckDB

```python
import duckdb

duckdb.sql("""
    SELECT type, height, ST_Area(geometry) as area
    FROM read_parquet(
        's3://overturemaps-us-west-2/release/2025-03-19.0/
         theme=buildings/type=building/*'
    )
    WHERE bbox.xmin > -97.8 AND bbox.xmax < -97.7
      AND bbox.ymin > 30.2  AND bbox.ymax < 30.3
    LIMIT 100
""")
```

<SpeakerNotes>
DuckDB reads GeoParquet directly from S3 — no download, no database server.

# What Doesn't Work Yet

- **No single model dominates** — GEO-Bench shows no FM consistently beats task-specific baselines
- **Geographic bias** — Prithvi v1.0 was US-only; Clay and DOFA have better global coverage but still patchy
- **Resolution loss** — MAE-based ViTs downsample 4–5x (224px input → 14×14 patches)
- **Mostly optical** — SAR and hyperspectral remain underserved (DOFA is the exception)
- **Supervised baselines still win** on well-labeled regional tasks
- **AI agents hallucinate** CRS and projection bugs — always verify spatial metadata

<SpeakerNotes>
Honest assessment. The hype cycle is real but so are the limitations.
GEO-Bench (Lacoste et al. NeurIPS 2023) is the most rigorous comparison so far.
Nature Machine Intelligence 2025 survey highlights geographic bias and compute
barriers as the top obstacles to operational deployment.
</SpeakerNotes>

# Two Paradigms for Self-Supervised Learning

<Columns cols={2} gap={5}>
<div>

### Contrastive Learning

Compare pairs of views of the **same** location:
- Augment a tile → two views
- Encode both → **pull embeddings together**
- Different tile → **push apart**

*Used by:* SatCLIP, GeoCLIP, CLIP-RS, Presto

</div>
<div>

### Masked Autoencoding

Reconstruct from **partial** observations:
- Mask 75% of image patches
- Encoder sees only visible 25%
- Decoder reconstructs full image

*Used by:* SatMAE, Prithvi, Scale-MAE, Clay, AlphaEarth

</div>
</Columns>

<SpeakerNotes>
Contrastive learning learns what makes tiles similar or different.
MAE learns spatial structure by predicting missing patches. They produce
different kinds of features — contrastive for retrieval, MAE for dense prediction.
</SpeakerNotes>

---

<Columns cols={2} gap={4} align="center" ratio="2fr 3fr">
<div>

## Contrastive Learning

Two augmented views of the **same** satellite tile get encoded. Their embeddings are pulled together in the space. Different tiles are pushed apart.

The model learns a **similarity metric** without any labels.

## Pixel vs Patch Embeddings

- **Pixel:** one vector per pixel — high spatial detail
- **Patch:** one vector per 16×16 block — richer semantics
- Pixel suits segmentation & change detection
- Patch suits classification & retrieval
- Most FMs moving toward **patch** with ViTs

</div>
<div>

<Video src="/videos/S10_PixelVsPatchEmbedding.mp4" title="Pixel vs Patch Embeddings" />

</div>
</Columns>

<SpeakerNotes>
An important design choice: pixel embeddings (Presto, SatCLIP) produce one
vector per pixel — great for dense tasks like segmentation. Patch embeddings
(SatMAE, Clay, DOFA) compress spatial info into one vector per patch — richer
semantics, better for classification and retrieval.
</SpeakerNotes>

---

# Pixel vs Patch Embeddings

| | **Pixel Embedding** | **Patch Embedding** |
|---|---|---|
| **Output shape** | One vector per pixel | One vector per patch (e.g. 16x16) |
| **Spatial detail** | High — preserves pixel-level info | Compressed — summarizes patch |
| **Best for** | Segmentation, change detection | Classification, retrieval, generation |
| **Context window** | Narrow — local pixel neighborhood | Wide — entire patch context |
| **Examples** | Presto, SatCLIP | SatMAE, Clay, DOFA, Prithvi |

> Most foundation models are moving toward **patch embeddings** with vision transformers — the spatial compression is worth the richer semantics.

# TorchGeo: Geospatial ML in 6 Lines

```python
from torchgeo.datasets import Sentinel2, EnviroAtlas
from torchgeo.samplers import RandomGeoSampler
from torch.utils.data import DataLoader

dataset = Sentinel2("data/") & EnviroAtlas("labels/")
sampler = RandomGeoSampler(dataset, size=256, length=1000)
dataloader = DataLoader(dataset, sampler=sampler, batch_size=16)
```

<SpeakerNotes>
The intersection operator is TorchGeo's key innovation. Standard PyTorch
DataLoader — just with geospatial awareness built in.

# Planetary Computer MCP

<Columns cols={2} gap={4} align="center">
<div>

**VS Code extension** — AI assistants meet Planetary Computer

- **Search** STAC catalogs from your editor
- **Browse** collections and items interactively
- **Generate** processing pipelines from natural language
- Works with **Claude, Copilot**, and any MCP agent

> Marketplace: **IsaacCorley.planetary-computer-mcp**

</div>
<div>

Built the extension in a weekend with Claude Code.

MCP lets AI agents call **real geospatial APIs** — not just generate code, but actually *execute* searches and preview results.

</div>
</Columns>

<SpeakerNotes>
The Model Context Protocol lets AI coding agents connect to external tools.
This extension gives Claude direct access to the Planetary Computer STAC API —
it can search, browse, and generate pipelines without leaving VS Code.
</SpeakerNotes>

## The Flywheel

**People → Models → Global Datasets → People**

- Active learning: human edits improve training data
- Open formats mean anyone can contribute and consume
- AI coding agents accelerate tool development
- Common schemas enable cross-dataset benchmarking

> Cloud-native data makes AI training easier. AI makes cloud-native tools better.

</div>
<div>

<Video src="/videos/S16_Flywheel.mp4" title="The GeoAI Flywheel" />

</div>

<SpeakerNotes>
This is the thesis: cloud-native geospatial and GeoAI aren't separate trends,
they're a feedback loop. Chris Holmes calls this the Global Spatial Data Ecosystem.
</SpeakerNotes>
